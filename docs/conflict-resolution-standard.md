# Korea Inside Conflict Resolution Standard

## 목적

충돌 상황에서
항상 같은 우선순위로 판단한다.

------------------------------------------------

Priority Order

------------------------------------------------

1.

현재 사용자 명시 지시 및 승인

Root `AGENTS.md`

Codex 작업 권한, 명시 승인, scope-control 최상위 기준

↓

2.

`PROJECT.md`

프로젝트 개요 기준이다.
Codex 실행 권한은 `AGENTS.md`를 따른다.

↓

3.

Product Vision / Product Constitution

제품 판단 기준이다.
파일 생성, 수정, 삭제, 이동, 이름 변경, 정리, commit, push 권한을 자동 승인하지 않는다.

↓

4.

↓

5.

↓

6.

↓

7.

Decision Rules

↓

10.

구현 편의성

구현이 쉽다는 이유로

상위 원칙이나 승인 범위를 변경하지 않는다.

------------------------------------------------

권한 해석 규칙

------------------------------------------------

문서 간 충돌 시
더 넓은 원칙보다
현재 사용자 명시 지시,
Root `AGENTS.md`,
승인된 작업 범위를 우선한다.

어떤 문서도
사용자 명시 승인 없이
파일 생성, 수정, 삭제, 이동, 이름 변경,
리팩토링, 포맷팅, stage, commit, push를
허용하지 않는다.

"좋아", "진행", "다음" 같은 표현은
현재 대화에서 승인된 범위 안에서만 해석한다.
보호된 작업은
사용자의 현재 명시 지시와 `AGENTS.md`를 먼저 확인한다.

------------------------------------------------

대표 충돌 사례

------------------------------------------------

SEO vs UX

→ 사용자 이해를 우선하면서 SEO 손실을 최소화한다.

Design vs Accuracy

→ 정확성을 우선한다.

AI 제안 vs 확정 결정

→ 확정 결정을 따른다.

새 기능 vs 기존 구조

→ 기존 구조 재사용을 우선 검토한다.

속도 vs 품질

→ 품질을 우선한다.

실물 사진 vs AI 이미지

→ 실물 정확성이 중요한 경우 사용자 제공 실물 사진을 우선한다.

------------------------------------------------

판단 절차

------------------------------------------------

1. 충돌 내용 정의

2. 영향 범위 분석

3. 현재 사용자 명시 지시와 승인된 작업 범위 확인

4. 우선순위 적용

5. 대안 비교

6. 권장안 제시

7. 사용자 명시 승인

8. 구현

------------------------------------------------

예외 규칙

------------------------------------------------

긴급 오류 수정도
사용자 명시 승인과 `AGENTS.md` 범위 통제 안에서만
간소화 절차를 적용할 수 있다.

단,
사후 변경 기록은
사용자가 승인한 범위 안에서만 남긴다.

------------------------------------------------

최종 원칙

------------------------------------------------

충돌은

AI가 임의로 해결하지 않는다.

현재 사용자 명시 지시,
Root `AGENTS.md`,
승인된 작업 범위를 확인하고

우선순위를 적용하고

사용자의 명시 승인을 받는다.
